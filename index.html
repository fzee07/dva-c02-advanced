Chapter 01

Advanced Lambda Patterns

Event-driven design, orchestration, and concurrency at scale

Book 2 covered Lambda fundamentals — concurrency, layers, versions, destinations. This chapter is about composing Lambda functions into real architectures: how to fan out work, coordinate multi-step processes, handle partial failures, and avoid the subtle traps that only appear at scale.

1.1 Synchronous vs Asynchronous vs Stream Composition

At an architectural level, how Lambda is invoked determines how you design for failure, retries, and ordering. The three models are not interchangeable:

ModelRetry BehaviourOrderingBest For
SynchronousCaller handles retryCaller-controlledAPIs, request/response
Asynchronous2 automatic retries, then DLQ/destinationNo orderingEvent reactions, S3/SNS triggers
Event source mapping (poll)Batch retried; stream blocks on failureOrdered per shard/groupKinesis, DynamoDB Streams, SQS
!Exam Trap · Stream Poisoning by a Bad Record

With Kinesis or DynamoDB Streams event source mappings, Lambda processes records in order per shard. If one record fails and you do not handle it, Lambda retries the entire batch and the shard is BLOCKED — nothing after that record processes until it succeeds or expires. Fix: enable "Bisect batch on function error" and configure a failure destination (on-failure SQS/SNS) so poison records are isolated instead of halting the stream.

1.2 Fan-Out and Fan-In

Fan-out distributes one event to many parallel workers. Fan-in aggregates many results back into one. These are the building blocks of parallel processing on AWS.

  • Fan-out: SNS → multiple SQS queues → multiple Lambda consumers, OR EventBridge → multiple targets
  • Fan-in: Workers write partial results to DynamoDB or S3; a final Lambda aggregates once all parts complete
  • Coordination challenge: knowing when all parallel work is done — this is where Step Functions Map state (Chapter 2) shines
</>Backend Dev AngleNode.js

A common fan-in pattern in Node.js: each worker increments an atomic counter in DynamoDB and writes its result. The worker that brings the counter to the expected total triggers the aggregation step:

const res = await ddb.send(new UpdateItemCommand({ TableName: "jobs", Key: { jobId: { S: jobId } }, UpdateExpression: "SET completed = completed + :one", ExpressionAttributeValues: { ":one": { N: "1" } }, ReturnValues: "UPDATED_NEW" })); if (Number(res.Attributes.completed.N) === expectedTotal) { await triggerAggregation(jobId); }

1.3 Idempotency

In event-driven systems, at-least-once delivery is the norm — the same event can arrive twice. Your Lambda must be idempotent: processing the same event twice produces the same result as processing it once.

  • Use an idempotency key (e.g., a unique event ID or order ID)
  • Store processed keys in DynamoDB with a conditional write (attribute_not_exists)
  • If the key already exists, skip processing — the event is a duplicate
  • AWS provides the Lambda Powertools idempotency utility to handle this automatically
!Exam Trap · SQS Standard + Lambda = Possible Duplicates

SQS Standard queues guarantee at-least-once delivery, not exactly-once. Combined with Lambda retries, your function WILL occasionally see duplicate messages. If processing is not idempotent (e.g., "charge the customer"), you will double-charge. The exam tests this: "payments processed twice" in an SQS Standard + Lambda design = missing idempotency, OR switch to SQS FIFO with content-based deduplication.

1.4 Lambda Powertools

AWS Lambda Powertools is a developer library (available for Node.js/TypeScript) that provides production-grade implementations of cross-cutting concerns:

UtilityWhat It Provides
LoggerStructured JSON logging with context injection
TracerX-Ray tracing with automatic segment capture
MetricsCustom CloudWatch metrics via Embedded Metric Format (EMF)
IdempotencyAutomatic idempotency using DynamoDB as the store
ParametersCached retrieval from SSM Parameter Store / Secrets Manager
Batch ProcessingPartial batch failure handling for SQS/Kinesis/DynamoDB

Chapter 02

AWS Step Functions

Orchestrating workflows with state machines

When a process has multiple steps, branching logic, parallel paths, retries, and human approvals, chaining Lambda functions manually becomes fragile. Step Functions is AWS's serverless orchestration service — you define a workflow as a state machine, and AWS manages execution, state, retries, and error handling.

2.1 Standard vs Express Workflows

FeatureStandardExpress
Max duration1 year5 minutes
Execution modelExactly-onceAt-least-once (async) / at-most-once (sync)
PricingPer state transitionPer execution + duration + memory
Execution historyFull, visible in consoleSent to CloudWatch Logs
Best forLong-running, auditable workflowsHigh-volume, short event processing
!Exam Trap · Standard vs Express by Volume and Duration

"High-volume event processing, thousands per second, each under a few seconds" = Express (cheaper at volume). "Order fulfillment workflow that may take days and needs an auditable history of every step" = Standard (exactly-once, full history, up to 1 year). Pricing models differ fundamentally: Standard charges per state transition, Express per execution time.

2.2 State Types

StatePurpose
TaskDo work — invoke a Lambda, ECS task, or another AWS service
ChoiceBranching logic based on input
ParallelRun multiple branches concurrently
MapRun the same steps for each item in an array (dynamic parallelism)
WaitPause for a duration or until a timestamp
Succeed / FailTerminate the execution with success or failure
PassPass input to output, optionally transforming it

2.3 Error Handling and Retries

Step Functions has built-in error handling that you would otherwise write by hand in Lambda. Each Task state supports:

  • Retry — retry the state on specific errors, with configurable interval, backoff rate, and max attempts
  • Catch — route to a fallback state when an error occurs (like a try/catch)
  • Timeout — fail the state if it runs too long
</>Backend Dev AngleNode.js

A Task state retrying a Lambda on transient errors with exponential backoff:

"ProcessPayment": { "Type": "Task", "Resource": "arn:aws:lambda:...:processPayment", "Retry": [{ "ErrorEquals": ["States.TaskFailed"], "IntervalSeconds": 2, "MaxAttempts": 3, "BackoffRate": 2.0 }], "Catch": [{ "ErrorEquals": ["States.ALL"], "Next": "RefundAndNotify" }] }

2.4 Orchestration Patterns

  • Saga pattern: a sequence of transactions where each has a compensating action. If step 3 fails, run compensations for steps 1 and 2 (e.g., refund payment, release inventory). Step Functions Catch states implement this cleanly.
  • Callback pattern (.waitForTaskToken): pause the workflow until an external system (or human) calls back with a task token. Used for human approval or async third-party processing.
  • Map state for dynamic parallelism: process an array of items in parallel without knowing the count in advance.
!Exam Trap · Step Functions vs SQS vs EventBridge

"Coordinate a multi-step workflow with branching, retries, and rollback" = Step Functions. "Decouple two services with a buffer" = SQS. "Route events by rules to multiple targets" = EventBridge. Step Functions is for ORCHESTRATION (one brain coordinating steps). The others are for CHOREOGRAPHY (services reacting independently).

Chapter 03

Advanced DynamoDB

Single-table design and access-pattern modeling

Book 2 covered DynamoDB mechanics — keys, indexes, streams, DAX. This chapter covers the hardest and most misunderstood topic: how to actually model data in DynamoDB. The relational instincts you bring from Postgres will work against you here.

3.1 The Single-Table Design Philosophy

In relational databases you normalize: one table per entity, joins at query time. In DynamoDB there are no joins, and queries must be fast at any scale. The advanced pattern is the opposite of relational thinking: store multiple entity types in a single table, structured so that the items you fetch together live together.

  • Design starts from access patterns, not entities. List every query your app will make FIRST.
  • Use generic key names: PK (partition key) and SK (sort key), not userId/orderId
  • Different entity types share the same table, distinguished by key prefixes
  • Related items share a partition key so a single query returns them together
iNote · Access Patterns Come First

You cannot add an efficient query to DynamoDB later the way you add an index in Postgres. If you did not design for an access pattern, your only options are a Scan (slow, expensive) or a new GSI. This is why DynamoDB modeling starts by exhaustively listing access patterns before a single key is chosen.

3.2 A Worked Example

Consider an e-commerce app. Access patterns: get a user, get a user's orders, get an order's items. In a single table:

PKSKEntityOther Attributes
USER#123PROFILEUsername, email
USER#123ORDER#1001Ordertotal, status, date
USER#123ORDER#1002Ordertotal, status, date
ORDER#1001ITEM#5Order Itemproduct, qty, price
ORDER#1001ITEM#6Order Itemproduct, qty, price

Query PK=USER#123 returns the profile and all orders in one call. Query PK=ORDER#1001 returns all items in that order. No joins, single-digit millisecond responses.

3.3 Index Overloading and Sparse Indexes

  • Index overloading: a single GSI serves multiple access patterns by storing different entity types with different meanings in the same GSI keys
  • Sparse index: a GSI only contains items that have the GSI key attribute. Items missing that attribute are not indexed — useful for querying a subset (e.g., only "active" records)
  • Inverted index: a GSI that swaps PK and SK, enabling the reverse lookup (find all orders containing a product)
!Exam Trap · Sparse Index for Filtered Queries

If you only want to query items in a particular state (e.g., orders with status=OPEN), add the GSI partition key attribute ONLY to those items. The GSI becomes a sparse index containing only open orders — querying it is far cheaper than scanning all orders and filtering. The exam frames this as "efficiently query only the small subset of items that match a condition."

3.4 Write Sharding for Hot Partitions

A hot partition occurs when too many reads/writes hit one partition key, exceeding its throughput. Solution: write sharding — append a calculated suffix (e.g., a number 1-10) to the partition key to spread writes across multiple partitions, then query all shards in parallel on read.

  • I know to list access patterns before designing keys
  • I can model multiple entity types in one table with shared partition keys
  • I understand index overloading, sparse indexes, and inverted indexes
  • I know what a hot partition is and how write sharding solves it

Chapter 04

Advanced API Gateway

Mapping templates, transformation, and custom domains

Book 2 covered API types, integrations, authorizers, and caching. This chapter covers the advanced REST API features: transforming requests and responses with VTL, custom domains, mutual TLS, and the integration patterns that avoid Lambda entirely.

4.1 Mapping Templates and VTL

In a non-proxy (custom) integration, API Gateway can transform the request before it reaches the backend and transform the response before returning it to the client — using Velocity Template Language (VTL) mapping templates.

  • Reshape a client JSON body into the format a legacy backend expects
  • Inject context (user ID from the authorizer, request time) into the backend request
  • Filter or rename fields in the response before returning to the client
  • Convert query parameters into a request body
iNote · Proxy vs Non-Proxy Trade-off

Lambda Proxy integration is simpler — the full request goes to Lambda and Lambda builds the full response. Non-proxy with VTL templates moves transformation INTO API Gateway, keeping Lambda focused on business logic. The cost: VTL is its own language, harder to test and debug. For most apps, proxy is preferred. Use VTL when integrating directly with AWS services without Lambda.

4.2 Direct AWS Service Integration

API Gateway can call AWS services directly — no Lambda needed. This removes the cost and cold-start of a Lambda for simple pass-through operations:

  • API Gateway → SQS: accept a request and drop it on a queue directly
  • API Gateway → DynamoDB: simple GetItem/PutItem without Lambda
  • API Gateway → Step Functions: start a workflow execution from an HTTP call
  • API Gateway → Kinesis: ingest records into a stream
!Exam Trap · Remove Lambda for Simple Pass-Through

If a question describes an API endpoint that simply forwards a request to SQS or writes a single item to DynamoDB — with no business logic — the cost-optimal answer is direct AWS service integration, not API Gateway → Lambda → service. You eliminate Lambda invocation cost and cold starts entirely.

4.3 Custom Domains and mTLS

  • Custom domain names: map api.yourcompany.com to your API Gateway, with an ACM certificate (regional, or us-east-1 for edge-optimized)
  • Base path mapping: route api.company.com/orders and api.company.com/users to different APIs under one domain
  • Mutual TLS (mTLS): require clients to present a certificate — used for B2B APIs and partner integrations where client identity must be cryptographically verified

  • I understand VTL mapping templates and when to use non-proxy integration
  • I know API Gateway can integrate directly with SQS, DynamoDB, Step Functions, Kinesis
  • I know when removing Lambda for direct integration is cost-optimal
  • I understand custom domains, base path mapping, and mTLS

Chapter 05

Event-Driven Architecture at Scale

EventBridge deep dive, choreography, and schema management

Event-driven architecture decouples producers from consumers — services emit events without knowing who consumes them. At scale, this requires careful thinking about routing, schema evolution, and the trade-off between choreography and orchestration.

5.1 EventBridge Deep Dive

FeatureDetail
Event BusDefault bus (AWS events), custom buses (your apps), partner buses (SaaS)
RulesMatch events by JSON pattern on any field; route to up to 5 targets each
Input TransformationReshape the event before delivery to a target
Archive & ReplayStore events and replay them later — for recovery or testing
Schema RegistryDiscover, store, and version event schemas; generate code bindings
Dead Letter QueueFailed event deliveries routed to SQS for inspection

5.2 EventBridge Pipes

EventBridge Pipes connects a source directly to a target with optional filtering and enrichment — a point-to-point integration without writing glue Lambda code. Source → (filter) → (enrich) → target.

  • Sources: SQS, Kinesis, DynamoDB Streams, Amazon MQ, Kafka
  • Enrichment: Lambda, Step Functions, API Gateway, API destinations
  • Targets: 15+ AWS services
  • Replaces the common "SQS → Lambda → transform → send somewhere" boilerplate
!Exam Trap · Pipes vs Bus

EventBridge Bus = one-to-many routing (fan-out to multiple targets by rules). EventBridge Pipes = point-to-point (one source to one target, with filter/enrich). "Connect a DynamoDB Stream to a Step Function with filtering, no custom code" = Pipes. "Route events to many subscribers by rules" = Bus.

5.3 Choreography vs Orchestration

AspectChoreographyOrchestration
ControlDistributed — services react to eventsCentral — one coordinator (Step Functions)
CouplingLooseTighter (coordinator knows the steps)
VisibilityHarder to trace the full flowFull execution history
Best forIndependent reactions, scaleMulti-step processes needing rollback
AWS serviceEventBridge, SNSStep Functions

5.4 Schema Evolution

As systems evolve, event formats change. The rule: never break existing consumers. Add new optional fields rather than renaming or removing existing ones. Version event schemas (e.g., detail-type "OrderPlaced.v2") when a breaking change is unavoidable, and let consumers migrate at their own pace.

  • I understand EventBridge Pipes and how it differs from a Bus
  • I can distinguish choreography from orchestration and choose between them
  • I understand backward-compatible schema evolution

Chapter 06

Advanced CloudFormation

Nested stacks, cross-stack references, and custom resources

Book 2 covered CloudFormation and SAM basics. At scale, a single template becomes unmanageable. This chapter covers structuring large infrastructure: nested stacks, cross-stack references, custom resources, and deployment safety.

6.1 Nested Stacks vs Cross-Stack References

ApproachHow It WorksBest For
Nested StacksA parent stack creates child stacks via AWS::CloudFormation::StackReusable components owned by one team, deployed together
Cross-Stack ReferencesStack A exports a value; Stack B imports it via Fn::ImportValueIndependently deployed stacks sharing resources (e.g., shared VPC)
!Exam Trap · You Cannot Delete an Exported Value in Use

If Stack A exports a value and Stack B imports it, you cannot delete or modify the export in Stack A while Stack B still imports it. CloudFormation blocks it to protect the dependency. This makes cross-stack references rigid — nested stacks are more flexible for tightly coupled resources. The exam tests this constraint directly.

6.2 Custom Resources

When CloudFormation does not natively support a resource or action, a Custom Resource lets you run a Lambda function during stack create/update/delete. CloudFormation calls your Lambda with the event type and waits for a success/failure signal.

  • Provision third-party resources (e.g., a Datadog monitor, a Cloudflare record)
  • Run setup logic (seed a database, populate config)
  • Look up values not available as CloudFormation parameters
  • Your Lambda MUST send a response to the pre-signed S3 URL or the stack hangs until timeout

6.3 Deployment Safety

  • Change Sets: preview exactly what will be added, modified, or deleted before executing
  • Stack Policies: protect specific resources from accidental updates during a stack update
  • DeletionPolicy / UpdateReplacePolicy: Retain or Snapshot critical resources (RDS, S3) on delete or replace
  • Drift Detection: detect manual changes made outside CloudFormation
  • Rollback Triggers: monitor CloudWatch alarms during deployment and auto-rollback if they fire
★Tip · StackSets for Multi-Account

CloudFormation StackSets deploy the same template across multiple AWS accounts and regions in one operation — essential for organizations using AWS Organizations. Common use: deploy a baseline security/IAM configuration to every account automatically.

Chapter 07

Advanced Security

ABAC, federation, resource policies, and cross-account access

Book 2 covered IAM, Cognito, KMS, Secrets Manager. This chapter goes deeper into the security patterns that appear in advanced DVA-C02 scenarios: attribute-based access control, identity federation, cross-account access, and automated secret rotation.

7.1 ABAC — Attribute-Based Access Control

RBAC (role-based) grants permissions by role. ABAC grants permissions based on tags (attributes) attached to identities and resources. Instead of writing a policy per team, you write one policy that allows access when the principal's tag matches the resource's tag.

</>Backend Dev AngleNode.js

An ABAC policy allowing a user to access only resources tagged with their own department:

{ "Effect": "Allow", "Action": "s3:*", "Resource": "*", "Condition": { "StringEquals": { "aws:ResourceTag/department": "${aws:PrincipalTag/department}" } } }

One policy scales to any number of departments — no policy changes when teams are added.

!Exam Trap · ABAC Scales, RBAC Multiplies

"A fast-growing company keeps creating new IAM policies every time a team is added" — the answer is ABAC. Tag principals and resources by attribute (team, project, environment) and write one tag-matching policy. RBAC requires a new role/policy per group; ABAC does not. This is the canonical ABAC exam scenario.

7.2 Identity Federation

Federation TypeUse CaseMechanism
SAML 2.0Corporate SSO (Active Directory, Okta)AssumeRoleWithSAML
Web Identity (OIDC)Social login, mobile appsAssumeRoleWithWebIdentity
Cognito Identity PoolsApp users needing AWS credentialsWraps web identity federation
IAM Identity CenterCentralized multi-account workforce accessSuccessor to AWS SSO

7.3 Resource Policies and Cross-Account Access

Identity-based policies attach to principals. Resource-based policies attach to resources (S3 buckets, Lambda functions, SQS queues, KMS keys) and specify who can access them — including principals in OTHER accounts.

  • Cross-account access needs BOTH: a resource policy granting the external principal, AND the external principal's identity policy allowing the action
  • S3 bucket policies, Lambda resource policies, and KMS key policies are the common cross-account surfaces
  • For role assumption across accounts: the trust policy (who can assume) + the permission policy (what they can do)
!Exam Trap · Cross-Account Needs Both Sides

Granting cross-account access is a two-sided handshake. The resource owner adds a resource policy (or role trust policy) allowing the external account. The external account adds an identity policy allowing its user/role to make the call. Miss either side and access fails. The exam loves "Account B cannot access Account A's bucket despite the bucket policy allowing it" — the answer is Account B's IAM identity policy is missing the permission.

7.4 Secrets Rotation

Secrets Manager rotates secrets using a Lambda rotation function that implements four steps: createSecret, setSecret, testSecret, finishSecret. For RDS, DynamoDB, and other supported databases, AWS provides the rotation Lambda; for custom secrets you write it.

★Tip · KMS Key Policies vs IAM

KMS is unusual: the key POLICY (resource-based) is the primary access control, not IAM. A KMS key policy must explicitly grant access — IAM permissions alone are not enough unless the key policy delegates to IAM. A common failure: "user has kms:Decrypt in IAM but still gets AccessDenied" — the key policy does not allow that principal.

Chapter 08

Performance Optimization

Caching, connection management, and tuning

Performance on AWS is an architecture decision, not an afterthought. This chapter covers the optimization levers DVA-C02 tests: caching layers, Lambda tuning, connection pooling, and DynamoDB performance.

8.1 Caching Layers

LayerServiceCaches
EdgeCloudFrontStatic and dynamic content at edge locations
APIAPI Gateway cacheAPI responses per stage
ApplicationElastiCache (Redis/Memcached)Database query results, sessions
DatabaseDynamoDB DAXDynamoDB reads (microsecond latency)
!Exam Trap · Match the Cache to the Bottleneck

DAX caches DynamoDB only. ElastiCache caches anything (RDS queries, sessions, computed results). API Gateway cache caches API responses. CloudFront caches at the edge. "Reduce repeated identical DynamoDB reads with no code change" = DAX. "Cache expensive RDS query results" = ElastiCache. Picking the wrong cache for the data source is a classic wrong answer.

8.2 Lambda Performance Tuning

  • Memory = CPU: Lambda CPU scales with memory. A function may run faster AND cheaper at higher memory because it finishes sooner. Use AWS Lambda Power Tuning to find the sweet spot.
  • Init outside the handler: initialize SDK clients and DB connections in the module scope, reused across warm invocations
  • Provisioned Concurrency: eliminate cold starts for latency-sensitive functions
  • SnapStart: for Java, snapshot the initialized environment to cut cold starts dramatically
  • Keep deployment packages small: smaller package = faster cold start
iNote · Higher Memory Can Be Cheaper

Counterintuitively, raising Lambda memory can REDUCE cost. Cost = (memory) x (duration). If doubling memory more than halves duration (because of the extra CPU), total cost drops. This is why blindly setting Lambda to 128MB to "save money" is often wrong. Power Tuning measures the real curve.

8.3 Connection Management

Lambda scales horizontally — each concurrent execution is a separate environment with its own connections. This breaks traditional connection pooling against RDS.

  • RDS Proxy: managed connection pooler between Lambda and RDS. Thousands of Lambda connections multiplex into a small RDS pool. Essential for Lambda + RDS at scale.
  • DynamoDB: no connection limit problem — it is HTTP-based, stateless. Prefer it for highly concurrent serverless workloads.
  • Reuse connections: open the connection in module scope, not inside the handler, so warm invocations reuse it.

8.4 DynamoDB Performance

  • Avoid hot partitions with high-cardinality partition keys or write sharding
  • Use eventually consistent reads (half the RCU cost) when strong consistency is not required
  • Use BatchGetItem / BatchWriteItem to reduce round trips
  • Use Query, never Scan, for predictable performance
  • DAX for read-heavy, repeated-read workloads

!Exam Trap · Lambda Is Not Always Cheapest

9.2 Data Transfer Costs

  • Savings Plans: flexible commitment discount across EC2, Fargate, Lambda

Chapter 10

Observability and Troubleshooting

Tracing, structured logging, and debugging distributed systems

When a request flows through API Gateway, Lambda, DynamoDB, and SQS, finding where it failed requires real observability. This chapter covers X-Ray deep, structured logging, and the troubleshooting patterns DVA-C02 tests.

10.1 X-Ray Deep Dive

  • Trace: the full path of one request across services
  • Segment / Subsegment: a service's work and its internal operations
  • Annotations: indexed — filter traces by them (e.g., userId)
  • Metadata: not indexed — extra debug context
  • Service Map: visual graph of services, latency, and error rates
  • Sampling: traces a subset (default 1 req/sec + 5%) to control cost
!Exam Trap · Active Tracing + SDK Both Needed

For full Lambda tracing you need BOTH: enable Active Tracing on the function, AND instrument the code with the X-Ray SDK (or Powertools Tracer) to capture downstream calls. Active Tracing alone traces only the invocation; the SDK traces calls to DynamoDB, S3, HTTP, etc. as subsegments.

10.2 Structured Logging

Plain text logs do not scale. Structured JSON logs let CloudWatch Logs Insights query by field. Include a correlation ID on every log line so you can trace one request across multiple Lambda invocations.

Chapter 11

Resilience and Error Handling

Retries, idempotency, circuit breakers, and DLQ strategy

Distributed systems fail partially and constantly. Resilience is designing so that those failures do not become outages or data corruption. This chapter covers the patterns DVA-C02 expects.

11.1 Retries and Backoff

  • Exponential backoff: increasing delay between retries (1s, 2s, 4s, 8s) to avoid hammering a struggling service
  • Jitter: add randomness to backoff so many clients do not retry in sync (the "thundering herd")
  • AWS SDK: retries with exponential backoff and jitter automatically for throttling/5xx errors
  • Max attempts: always cap retries — infinite retries amplify outages
!Exam Trap · Retries Need Idempotency

Retrying a non-idempotent operation (e.g., "create order", "charge card") can create duplicates. Before adding retries to any write operation, ensure it is idempotent — typically via an idempotency key stored with a conditional write. Retries + non-idempotent writes = data corruption at scale.

11.2 Dead Letter Queues and Failure Destinations

MechanismApplies ToCaptures
Lambda DLQAsync LambdaEvents that failed after all retries (SQS/SNS only)
Lambda DestinationsAsync LambdaBoth success and failure outcomes (4 target types)
SQS DLQSQS queuesMessages exceeding maxReceiveCount
Step Functions CatchWorkflowsRoutes failed states to a handler/compensation

11.3 Circuit Breaker Pattern

When a downstream dependency is failing, continuing to call it wastes resources and worsens the problem. A circuit breaker tracks failures and "opens" after a threshold — short-circuiting calls and failing fast — then periodically tests if the dependency has recovered before "closing" again.

11.4 Decoupling for Resilience

  • Put a queue (SQS) between producer and consumer so a slow/down consumer does not break the producer
  • Use async patterns so a failure in one path does not cascade
  • Set timeouts on every external call — never wait indefinitely
  • Design for graceful degradation: serve cached/partial data when a dependency is down

When should you choose Step Functions Standard over Express?

Answer

Standard for long-running (up to 1 year), exactly-once, auditable workflows needing full execution history — e.g., order fulfillment over days. Express for high-volume, short (under 5 min) executions where per-execution pricing is cheaper — e.g., thousands of event-processing runs per second. Standard is priced per state transition; Express per execution time.

Question 2

Explain the saga pattern and how Step Functions implements it.

Answer

A saga is a sequence of transactions where each step has a compensating action that undoes it. If a later step fails, the compensations for completed steps run in reverse to restore consistency (e.g., refund payment, release inventory). Step Functions implements this with Catch states routing failures to compensation states.

Question 3

Fill in the blank: A ______ index in DynamoDB includes only items that possess the index's key attribute, making it efficient to query a subset of items.

Answer

sparse. By adding the GSI key attribute only to items you care about (e.g., status=OPEN), the index contains only that subset, so querying it avoids scanning all items.

Question 4

Why can raising Lambda memory sometimes reduce total cost?

Answer

Lambda CPU scales with allocated memory. Cost = memory x duration. If more memory (more CPU) makes the function finish more than proportionally faster, the reduced duration outweighs the higher per-ms price, lowering total cost. AWS Lambda Power Tuning finds the optimal point.

Question 5

What is the difference between an EventBridge Bus and EventBridge Pipes?

Answer

A Bus does one-to-many routing: events are matched by rules and sent to multiple targets (fan-out). Pipes is point-to-point: a single source connects to a single target with optional filtering and enrichment, replacing glue Lambda code. Bus = routing hub; Pipes = a smart connector.

Question 6

Why is KMS access control different from most AWS services?

Answer

For most services, IAM identity policies are the primary control. For KMS, the KMS KEY POLICY (resource-based) is primary — a principal needs the key policy to grant access (directly or by delegating to IAM). Having kms:Decrypt in IAM is insufficient if the key policy does not allow that principal, producing AccessDenied.

Question 7

Fill in the blank: Cross-account access requires both a ______ policy on the resource and an ______ policy on the calling principal.

Answer

resource; identity. The resource owner grants the external principal via a resource (or trust) policy, and the caller's account grants its principal the action via an identity policy. Both sides are mandatory.

Question 8

How do you prevent a single poison record from blocking a Kinesis/DynamoDB stream processed by Lambda?

Answer

Enable 'bisect batch on function error' so Lambda splits the failing batch to isolate the bad record, and configure an on-failure destination (SQS/SNS) to capture it. This stops the poison record from blocking the shard while preserving it for inspection.

Question 9

What is ABAC and why does it scale better than RBAC for a fast-growing organization?

Answer

ABAC (Attribute-Based Access Control) grants permissions by matching tags on principals and resources rather than assigning roles. One tag-matching policy (e.g., allow when PrincipalTag/team equals ResourceTag/team) covers any number of teams. RBAC requires a new role/policy each time a team is added; ABAC requires only tagging, so it scales without policy sprawl.

Question 10

Why must a write operation be idempotent before adding retries, and how do you make it idempotent?

Answer

Retries can resend the same request, so a non-idempotent write (create order, charge card) executes multiple times, causing duplicates or double-charges. Make it idempotent with an idempotency key: store the key on first processing using a conditional write (attribute_not_exists). If the key already exists, the operation is a duplicate and is skipped, so repeated processing has no additional effect.

Part 4 — Scenario-Based Questions (4 Questions)

Advanced scenarios. Choose the optimal approach.

Scenario 1

A SaaS company processes tenant data and keeps creating new IAM policies as customers onboard, now hitting policy limits. Each tenant's resources are tagged with a tenantId. They need a scalable access model with minimal ongoing policy management.

  • ACreate one IAM role per tenant
  • BUse ABAC with a policy matching PrincipalTag/tenantId to ResourceTag/tenantId
  • CMake all resources public with application-level checks
  • DUse a single shared admin role
Correct Answer: B

ABAC with tag matching is the canonical solution for multi-tenant scale. One policy covers all tenants; onboarding a tenant requires only tagging, no new policies — eliminating policy sprawl and limit issues.

Scenario 2

An e-commerce checkout must: charge payment, reserve inventory, then create a shipment. If shipment creation fails, the payment must be refunded and inventory released. The process must be auditable and may take minutes. What architecture is best?

  • AA single large Lambda with try/catch
  • BSQS chain between three Lambdas
  • CStep Functions Standard with Catch states implementing the saga pattern
  • DEventBridge rules triggering each step
Correct Answer: C

Step Functions Standard provides auditable, exactly-once orchestration with Catch states for compensating rollback (refund, release inventory) — the saga pattern. A single Lambda is fragile; SQS/EventBridge are choreography without central rollback coordination.

Scenario 3

A serverless API on Lambda + DynamoDB sees flash-sale spikes where the same few hot product items are read tens of thousands of times per second, causing throttling. The team wants the fix with the least code change and lowest added latency.

  • ASwitch DynamoDB to provisioned with huge RCUs
  • BAdd DynamoDB DAX in front of the table
  • CRe-architect reads to ElastiCache Redis
  • DAdd read replicas
Correct Answer: B

DAX is DynamoDB-API-compatible (point the SDK at the DAX endpoint — minimal code change) and serves hot-item reads from memory at microsecond latency, absorbing the spike. ElastiCache requires significant code change; DynamoDB has no read replicas; over-provisioning RCUs is costly and still risks hot-partition limits.

Scenario 4

A Lambda in a private subnet downloads large files from S3 through a NAT Gateway. The monthly bill shows large NAT Gateway data processing charges. The team wants to cut this cost without changing application logic.

  • AMove the Lambda to a public subnet
  • BAdd an S3 Gateway VPC Endpoint and route S3 traffic through it
  • CEnable S3 Transfer Acceleration
  • DReplace the NAT Gateway with a NAT instance
Correct Answer: B

An S3 Gateway VPC Endpoint routes S3 traffic privately within the VPC, bypassing the NAT Gateway entirely — eliminating its per-GB data processing charges. The endpoint is free and requires no application change. A public subnet weakens security; Transfer Acceleration adds cost; a NAT instance still processes the data.
